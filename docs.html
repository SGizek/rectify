Rectify usage guide + Rappad internal API reference

Rectify

Rectify is an unofficial multi-purpose tool for Rappad. It communicates with Rappad's internal API through a server-side proxy, bypassing CORS restrictions. No data is saved — sessions exist only in memory and are cleared when the tab closes.

How it works

Your browser talks to the Rectify proxy (/api/proxy), which makes requests to Rappad server-side using a per-session cookie jar. This means Rappad sees a normal authenticated request and CORS is never an issue.

Sessions are identified by a UUID stored in sessionStorage. Closing the tab ends the session. Nothing is written to a database or disk.

Proxy endpoint

All actions go through a single endpoint:

POST /api/proxy
Content-Type: application/json

{ "action": "...", "sessionId": "uuid", ... }

The sessionId is auto-generated by the frontend and must be included in every request. The csrfToken returned on login must be passed with any action that modifies data.

Rappad Internal API

Rappad does not have a public API. These endpoints were discovered by inspecting network requests in browser DevTools. They are undocumented and unofficial — use at your own risk. All endpoints are on https://www.rappad.co.

All state-changing requests require a valid x-csrf-token header and an active session cookie. The CSRF token is returned on login/register and changes each session.

GET/users/{username}

Returns the full HTML profile page for a user. Not a JSON endpoint — returns HTML. Rappad sets X-Frame-Options: sameorigin so direct iframing is blocked.

Notes

All POST/PUT requests must include these headers:

x-csrf-token: {token from login}
x-requested-with: XMLHttpRequest
content-type: application/x-www-form-urlencoded; charset=UTF-8

The CSRF token is tied to your session. If your session expires, re-login to get a fresh token. Rappad is built on Ruby on Rails — the CSRF system is standard Rails authenticity_token under the hood.
